1. 60-second elevator pitch

"I built an end-to-end automation framework against automationexercise.com using Playwright + Python + pytest. The architecture follows the Page Object Model, with separate layers for pages, components, flows, utilities, and configuration. Tests are data-driven from YAML files, configured via typed Pydantic settings, and report into Allure with full evidence per test — Playwright trace, video, logs, and screenshots. The full suite runs on every push to main via GitHub Actions, and credentials are protected with .env + optional SOPS + age encryption and a gitleaks pre-commit hook. I also layered a project-level AI workflow on top — Cursor rules, MCP servers, and an AGENTS playbook — so any AI assistant can extend it safely."

Key principle: 4 horizontal layers (pages → components → flows → tests) and 1 vertical (utils + config). Tests never import Playwright directly; they call flows, which call pages, which use components and locators. This keeps every layer focused on one job.

5. OOP — what & why

Object-Oriented Programming, plain English

OOP organises code into classes (blueprints) and objects (instances). Each class bundles data (attributes) and behavior (methods) related to one concept. Four pillars:

PillarPlain EnglishWhere in our project
EncapsulationGroup related data + behavior; hide internals.LoginPage hides selectors as class constants; tests never see them.
InheritanceRe-use base behavior in child classes.Every page extends BasePage, inheriting self.page, self.log, and helpers like open().
PolymorphismSame method name behaves differently per class.Each page can override is_loaded() to define its own readiness check.
AbstractionHide complexity behind a simple interface.Tests call login_flow(page); they don't care about cookie banners, ad popups, WebAuthn dialogs, etc.

Why OOP in tests?

  • Selectors live in one file per page → if the UI changes, you change one place.
  • Tests read like English (cart.add(product) instead of page.locator('[data-id=...]').click()).
  • Inheritance lets every page share logging, screenshot helpers, ad-blocking — without copy/paste.

6. POM — Page Object Model

Definition: Each web page (or major component) is represented by a Python class. The class exposes user-level actions and state queries, never raw selectors.

Anatomy of a page object

class LoginPage(BasePage):
    URL = "/login"
    EMAIL_INPUT = '[data-qa="login-email"]'
    PASSWORD_INPUT = '[data-qa="login-password"]'
    LOGIN_BUTTON = '[data-qa="login-button"]'
    LOGIN_ERROR = "form p:has-text('incorrect')"

    def login(self, email: str, password: str) -> "LoginPage":
        self.page.fill(self.EMAIL_INPUT, email)
        self.page.fill(self.PASSWORD_INPUT, password)
        self.page.click(self.LOGIN_BUTTON)
        return self

    def is_authenticated(self) -> bool:
        return self.page.locator("a:has-text('Logout')").is_visible()

Why this works

  • Locators are constants at the top, easy to find and change.
  • Methods return self when chainable, supporting fluent style: LoginPage(page).login(...).logout().
  • State queries (is_authenticated, is_on_login_page) make assertions readable.
  • BasePage centralises wait helpers, logger, screenshot capture.

Components vs pages

A component is a re-usable widget that appears on multiple pages — e.g. the site header, a product card, the price filter. We extracted them so every page that uses them gets the same logic for free.

7. Data-Driven testing

"Same test logic, many input rows." We define inputs in data/queries.yaml:

queries:
  - name: tshirt
    term: "T-Shirt"
    min_price: 400
    max_price: 700
  - name: jeans
    term: "Jeans"
    min_price: 500
    max_price: 1500
  - name: dress
    term: "Dress"
    min_price: 0
    max_price: 9999

Then test_search_data_driven.py uses pytest's parametrize:

@pytest.mark.parametrize("query", load_queries(), ids=lambda q: q["name"])
def test_search_with_price_filter(logged_in_page, query):
    ...

Why data-driven?

  • Add a new test scenario by editing YAML — no Python change.
  • Each row appears as a separate test in the Allure report (test_search[tshirt], test_search[jeans]).
  • Non-developers (PMs, business analysts) can review/extend the dataset.

9. Playwright deep dive

Why Playwright over Selenium / Cypress?

  • Auto-waiting — every action waits for the element to be actionable. Fewer flaky sleep()s.
  • One driver, three browser engines — Chromium, Firefox, WebKit. No driver mismatch hell.
  • Built-in tracing — captures DOM snapshots, network, console, screenshots in one zip; viewable with npx playwright show-trace.
  • Network interception — block ads, mock APIs, throttle.
  • Storage state — save/restore cookies + localStorage, skipping repeated logins.
  • Same API in JS, TS, Python, Java, C#.

Locator strategy (in order of preference)

  1. page.get_by_role("button", name="Login") — accessibility-first, robust.
  2. page.get_by_test_id("login-button") — explicit hooks via data-test.
  3. page.get_by_text("Welcome") — user-visible text.
  4. CSS attribute: [data-qa="login-button"].
  5. XPath — last resort. Mandated by the brief for product card collection in SearchResultsPage.

Key Playwright concepts in this project

ConceptWhat we use it for
browser.new_context()Fresh, isolated browser session — like a private window.
context.tracing.start()Records every action for the debugger trace viewer.
context.route()Block ad domains at the network layer.
page.add_init_script()Run JS before any page script — used to override navigator.credentials so WebAuthn never prompts.
page.add_locator_handler()Auto-dismiss popups whenever they appear (cookie banner, etc.).
storage_statePersisted login session for the shared-browser fixture.
recordVideoPer-test WebM video evidence.

10. pytest deep dive

Core ideas

  • Test discovery — pytest finds files matching test_*.py, classes Test*, functions test_*.
  • Assertions — plain assert. pytest rewrites them to give rich diff output.
  • Fixtures — dependency injection: a function declares def test_x(page): and pytest resolves page automatically.
  • Markers — tags like @pytest.mark.smoke; runnable via pytest -m smoke.
  • Parametrize — same test, many inputs.
  • Hooks — extension points (pytest_runtest_makereport, pytest_collection_modifyitems, etc.).

Our pytest.ini highlights

addopts =
    -v --strict-markers --tb=short --color=yes
    --alluredir=reports/allure-results
    --clean-alluredir
    --junit-xml=reports/junit.xml
    --maxfail=3                   ← stop after 3 failures (fail-fast)

markers =
    e2e: full end-to-end purchase flow
    smoke: fast sanity checks
    search, cart, data_driven: scope filters

log_cli = true                    ← live logs in console
log_file = reports/pytest.log     ← also a log file

11. Fixtures & conftest.py

Fixtures are the killer feature of pytest. They:

  • Set up resources (browser, page, login session).
  • Tear them down automatically.
  • Have scopes: function (default), class, module, session.
  • Inject into tests by parameter name.

Scopes by example

ScopeLifetimeUsed for
functionRecreated for each test.Per-test evidence dir, per-test page (when not shared).
sessionOne instance for the whole pytest run.Settings, the shared logged-in browser.

Hooks we use in conftest.py

  • pytest_runtest_makereport — capture final-state screenshot + Allure attachment after each test.
  • pytest_runtest_logfinish — arm the watchdog timer so the process can't hang.
  • pytest_terminal_summary — print the report path at the end.

12. Session-scoped shared browser

Problem: 5 tests, each doing its own UI login = 5 × 8 seconds = wasted minutes.

Solution: One session-scoped fixture (_shared_browser_page) opens the browser once, performs a single visible login, and yields the same Page to every test that asks for logged_in_page. The browser stays open the entire suite.

The flow

  1. test_login.py — runs in its own dedicated browser (because it tests the auth flow itself: negative → positive → logout).
  2. _shared_browser_page opens a second browser, logs in once.
  3. test_e2e_purchase_flow.py + 3 test_search_data_driven rows all reuse it.
  4. Session ends → trace stops, video finalises, browser closes.

Result: full suite went from 232s → ~96s, with continuous video for the demo.

13. Allure reporting

Allure is a dual-piece tool:

  1. allure-pytest (Python) — writes a JSON file per test result + attachments into reports/allure-results/.
  2. allure CLI (Node.js, installed via npm i -g allure-commandline) — converts JSON into an interactive HTML site.

Customisations we added

File / DecoratorEffect
environment.propertiesShows OS, browser, base URL, Python version on the report's Overview tab.
categories.jsonAuto-buckets failures (e.g. "Network issues", "Locator not found").
@allure.title("...")Friendly test title in the UI (instead of test_login_negative_then_positive).
@allure.severity(critical)Lets reviewers filter by importance.
@allure.tag("smoke", "auth")Multi-dimensional filtering.
@allure.step("Click Login")Each step shows in the timeline with screenshots/logs attached.
allure.attach(...)Attach screenshots, videos, JSON summaries, trace zips.
Why allure open, not double-clicking index.html? Browsers block file:// requests for the JSON data the report loads (CORS). allure open serves it via a tiny HTTP server.

15. GitHub Actions CI

File: .github/workflows/e2e-tests.yml. Triggers on push/PR to main. Runs on ubuntu-latest.

Pipeline steps

  1. Checkout code.
  2. Setup Python 3.11.
  3. Cache pip + Playwright browsers.
  4. pip install -e . + playwright install --with-deps chromium.
  5. Run test_login.py in its own job.
  6. Run the rest of the suite (shared-browser tests).
  7. Build Allure HTML report.
  8. Upload reports/ as a GitHub artifact (test-reports, retained 30 days).

Secrets

Credentials live in GitHub Secrets (SITE_EMAIL, SITE_PASSWORD) and are injected as env vars at runtime. They never appear in code, logs, or artifacts.

Headed vs headless

CI sets HEADED: "false" — Linux runners have no display. Local devs run headed for the demo.

16. Secrets: .env, SOPS+age, gitleaks

Three layers

  1. .env (gitignored) — plain text, local only. Loaded by python-dotenv.
  2. SOPS + age — optional. Encrypt .env at rest in the repo as secrets/.env.sops. Decrypts at runtime with the user's age key.
  3. gitleaks pre-commit hook — blocks commits that contain secrets (regex-matched).

What is SOPS?

"Secrets OPerationS" — Mozilla's tool for encrypting structured files (.env, JSON, YAML). Keeps the file structure, encrypts only the values, so diffs stay readable.

What is age?

"Actually Good Encryption" — a small, modern asymmetric encryption tool by Filippo Valsorda. Much simpler than GPG. SOPS uses age as the key backend.

Why both gitleaks and SOPS?

Defense in depth. SOPS prevents accidental commits of plaintext secrets. gitleaks is the safety net — it scans regardless and blocks commits with anything that looks like a credential.

17. AI tooling layer (MCP, Cursor rules, AGENTS.md)

MCP — Model Context Protocol

An open standard for connecting LLMs to external tools/data. Two MCP servers wired into this project:

  • Playwright MCP — lets an AI agent drive the browser directly (record, replay, inspect).
  • Filesystem MCP — gives the agent scoped file read/write access.

Cursor project-level rules

Files under .cursor/rules/*.mdc. Each rule defines an "always-on" instruction or pattern (e.g. "never put hard-coded waits", "prefer get_by_role"). Cursor injects them into every prompt automatically.

Slash commands

.cursor/commands/*.md — reusable prompts the user can trigger with /command-name, e.g. /add-page scaffolds a new POM class with the right boilerplate.

Personas

docs/personas/*.md — role descriptions (QA Architect, SDET, AI Reviewer) the user can invoke to shift the AI's tone and priorities.

AGENTS.md

The single source of truth for any AI assistant working in the repo: project conventions, naming rules, locator policy, security policy, what to do / never do. Universal — works with Cursor, Claude, Copilot, Aider, etc.

18. Robustness tricks

ProblemSolution
Playwright teardown hangs on Windows.Watchdog timer — ctypes.windll.kernel32.TerminateProcess after WATCHDOG_GRACE_S seconds.
Cookie / ad / WebAuthn popups disrupt flow.page.add_locator_handler for popups; page.add_init_script overrides navigator.credentials; context.route blocks ad domains.
Page screenshots taken before render._wait_for_page_ready() waits for domcontentloaded, load, then networkidle with timeouts before capture.
Stale Chromium / pytest leftovers from previous runs.Pre-flight kill — but only processes whose command line contains ms-playwright, so we never kill the user's regular Chrome / Teams / Edge.
Test fails halfway, blocking the rest.--maxfail=3 stops the run early so feedback is fast.
Network is slow / flaky.Generous, explicit timeouts in locators (timeout=8_000) instead of fixed sleeps.
Decimal money math.decimal.Decimal in price_parser.py — never float.

19. The 5 tests & 35 assertions

1. test_login.py (9 assertions)

  • Negative: wrong password → error visible, still on /login, not authenticated.
  • Positive: correct creds → redirected, "Logged in as" header visible, Logout link visible.
  • Logout: back to /login, not authenticated.

2. test_e2e_purchase_flow.py (~14 assertions)

  • Search executed, results visible.
  • Price filter applied (client-side, XPath-mandated card collection).
  • Add a random variant of each filtered product to cart.
  • Cart shows correct line items, quantities, prices.
  • Cart total matches sum of items (Decimal math).
  • Proceed to checkout — order summary page asserts: row count, addresses block, total + tax line.

3–5. test_search_data_driven.py (3 rows × ~4 assertions)

  • Each YAML row drives one parametrized test.
  • Asserts results visible, filtered count > 0, all prices within range.

Total: 35 assertions across 5 tests, all green.

20. Likely interview questions & concise answers

Q: Walk me through the project.
Use the elevator pitch in section 1, then offer to drill into any layer.
Q: Why Playwright over Selenium?
Auto-waiting kills 90% of flakiness, built-in tracing/video out of the box, single API across 3 browser engines, fast (1 process, 1 protocol via CDP). Selenium needs separate drivers and lots of explicit waits.
Q: Explain POM.
Each page in the SUT is a Python class. Selectors are class constants. Methods expose user-level actions (login, add_to_cart) and state queries (is_authenticated). Tests never touch raw selectors → maintainable.
Q: How is your framework data-driven?
YAML files in data/ hold input rows. pytest.mark.parametrize spreads them into separate tests. Adding a row needs zero code change.
Q: How do you avoid logging in 5 times?
Session-scoped fixture _shared_browser_page: one browser, one UI login, every logged_in_page test reuses the same page. Cut runtime from 232s to 96s.
Q: What if the UI changes?
Selector changes touch one POM class. Behavior changes touch one flow. Tests stay readable.
Q: How do you handle flakiness?
Auto-waiting locators with explicit timeouts, ad/popup handlers, network blocking, page-ready waits before screenshots, pytest-rerunfailures for the genuinely flaky ones.
Q: How do you keep secrets out of git?
Three layers: .env in .gitignore, optional SOPS+age encryption, gitleaks pre-commit hook as the safety net. CI uses GitHub Secrets.
Q: How is the report generated?
allure-pytest writes JSON per test → allure CLI (Node.js) builds static HTML → I serve it with allure open because file:// would hit CORS.
Q: Why Decimal for money?
Floats lose precision (0.1 + 0.2 == 0.30000000000000004). Cart totals must be exact, so we parse strings into Decimal.
Q: Did you try parallel execution?
Yes, with pytest-xdist. Slower for headed UI tests due to xdist overhead and multiple browser instances on the same display. Reverted; would revisit if we move to headless containerised runs.
Q: How does CI work?
GitHub Actions on every push to main: setup Python, install deps, install Chromium, run tests, build Allure HTML, upload reports/ as a 30-day artifact. Credentials via GitHub Secrets, headless on Linux.
Q: What's MCP?
Model Context Protocol — an open standard for plugging tools into LLMs. We wired Playwright MCP and Filesystem MCP so any AI assistant can drive the browser and read repo files safely.
Q: What's conftest.py?
A pytest "magic" file. Fixtures and hooks defined there are auto-discovered by every test in the same folder (and subfolders) — no explicit import needed.
Q: Difference between browser, context, page in Playwright?
Browser = the running Chromium process. BrowserContext = isolated session (cookies, storage) — like an incognito window. Page = a tab inside a context. Multiple pages can share a context (same login state); contexts are independent.
Q: How do you handle WebAuthn / Passkey prompts?
page.add_init_script overrides navigator.credentials.create and navigator.credentials.get to throw, so the site falls back to the password form before any prompt appears.
Q: What's a fixture scope?
How long the fixture instance lives: function (default — recreated per test), class, module, or session (one per pytest run). We use session for the shared browser.

21. Gotchas & trade-offs (be honest about them)

  • Headed runs are slower — they're for the demo / debugging. CI is headless.
  • Shared browser couples tests — if one test corrupts state, the next can fail. Mitigated by clear flow methods and assertions on entry.
  • Watchdog kills the process hard — sometimes loses the last log line. Acceptable trade-off vs hangs.
  • SOPS adds friction for new contributors — that's why it's optional, with a plain .env fallback.
  • Allure CLI is Node.js — yet another runtime to install. Could ditch it for a Python-only reporter (e.g. pytest-html), but Allure UX is unmatched.
  • YAML for data is great until schemas grow — at scale, would migrate to typed dataclasses or JSON Schema validation.
