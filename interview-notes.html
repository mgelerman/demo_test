3. Interview Q&A bank

Q: Why Playwright over Selenium?
Auto-waiting kills 90% of flakiness, built-in tracing/video out of the box, single API across 3 browser engines, fast (1 process, 1 protocol via CDP). Selenium needs separate drivers and lots of explicit waits.
Q: How do you keep secrets out of git?
Three layers:
  1. .env in .gitignore
  2. Optional SOPS + age encryption
  3. gitleaks pre-commit hook as the safety net
CI uses GitHub Secrets.
Q: Difference between browser, context, page in Playwright?
Browser = the running Chromium process.
BrowserContext = isolated session (cookies, storage) — like an incognito window.
Page = a tab inside a context.
Multiple pages can share a context (same login state); contexts are independent.

4. Core concepts (one-liners)

  • pytest bottom line: Playwright automates the browser; pytest runs the tests and judges pass/fail.
  • POM: Page Object Model — separate layers for pages, components, flows, utilities, configuration.
  • Data-driven: tests read inputs from YAML files, configured via typed Pydantic settings, reporting into Allure with full evidence (trace, video, logs, screenshots).

5. Pydantic Settings (typed configuration)

Location: c:\demo_test\src\config\settings.py

Pydantic is a Python library for data validation. pydantic-settings extends it to load configuration from environment variables and .env files into a typed class.

Benefits

ProblemPydantic solves it
Forgot to set SITE_EMAIL?Crashes immediately with a clear error, not halfway through a test.
Set HEADED=yes instead of true?Validates and converts types automatically.
Typo in variable name?IDE autocomplete works because it's a real class with real attributes.
Where do settings come from?One place to look — the Settings class.

How it loads values (priority order)

  1. Environment variables (e.g. set in CI via GitHub Secrets)
  2. .env file in project root (local dev)
  3. Default values in the class definition

7. GitHub Actions (CI/CD)

GitHub Actions is GitHub's built-in CI/CD system. It automatically runs tasks when something happens in the repo — e.g. push or pull request.

What happens on every push to main

  1. GitHub spins up a fresh Linux VM
  2. Checks out the code
  3. Installs Python, deps, Playwright browsers
  4. Runs all 5 tests headless (no display on Linux)
  5. Builds the Allure HTML report
  6. Uploads reports as a downloadable artifact (30-day retention)

File: .github/workflows/e2e-tests.yml

Why it matters

  • Catches bugs before they reach production
  • No "works on my machine" excuses — clean environment
  • Reports always available, even if a dev forgets to run locally

9. SOPS + age — encryption flow

  1. Run scripts/encrypt-env.ps1
  2. SOPS encrypts .env → secrets/.env.sops (safe to commit)
  3. The file looks like: 
    SITE_EMAIL=ENC[AES256_GCM,data:abc123...,tag:xyz...]
SITE_PASSWORD=ENC[AES256_GCM,data:def456...,tag:uvw...]
  4. At runtime, if .env doesn't exist, the framework decrypts secrets/.env.sops using your local age key.

Why "optional": adds friction for new contributors (need age keys). Plain .env + .gitignore is enough for this demo. SOPS is there for production-grade security if needed.

10. Manual gitleaks installation (Windows)

  1. Releases page: https://github.com/gitleaks/gitleaks/releases
  2. Download gitleaks_X.X.X_windows_x64.zip (or _arm64 for ARM)
  3. Extract → you get gitleaks.exe
  4. Move to a folder in PATH, e.g. C:\Users\MOSHEGEL\AppData\Local\Microsoft\WindowsApps\ or create C:\tools\ and add to PATH
  5. Verify: gitleaks version

After this, pre-commit will find the existing gitleaks and won't try to download it.

11. TOML

TOML stands for Tom's Obvious Minimal Language. A configuration format like YAML/JSON, designed to be easy to read, easy to write, unambiguous.

TOML vs YAML vs JSON

FormatStrengthsWeaknesses
TOMLClean [headers], no indentation traps, great for configLess common, not ideal for deeply nested data
YAMLVery readable, great for data files, widely usedIndentation-sensitive, surprising edge cases (yes becomes true)
JSONUniversal, strict, every language supports itVerbose, no comments allowed

Where TOML is used in this project

FilePurpose
pyproject.tomlPython project metadata, dependencies, tool configs (ruff, black, mypy) — modern PEP 621 standard

Why Python adopted TOML

Before TOML, Python projects had multiple config files:

  • setup.py — package metadata
  • requirements.txt — dependencies
  • setup.cfg — more metadata
  • .flake8, .isort.cfg, … — per-tool configs

pyproject.toml replaces all of them in one file.

12. Where Node.js comes in

We don't write Node.js code, but we depend on one Node tool: allure-commandline (installed globally via npm i -g allure-commandline). It generates the static HTML Allure report from JSON results. 13. Dev / quality tools

Ruff — fast Python linter

Replaces flake8 + isort + pyupgrade in one Rust-powered tool (10–100× faster).

Black — formatter

Deterministic style — same input always → same output.

# Before Black:
x = {  'a':1,'b':2 }
def foo(x,y,z): return x+y+z

# After Black:
x = {"a": 1, "b": 2}
def foo(x, y, z):
    return x + y + z

15. Watchdog

A safety timer that forcefully kills the test process if it hangs too long.

The problem

Playwright on Windows sometimes hangs during teardown — browser closes, but the Python process sits doing nothing for 5–10+ minutes.

How it works

  1. Timer starts after each test (pytest_runtest_logfinish hook)
  2. Grace period of ~8 seconds (_WATCHDOG_GRACE_S)
  3. If still alive → force kill via Windows TerminateProcess API

17. OOP — what & why

OOP organises code into classes (blueprints) and objects (instances). Each class bundles data (attributes) and behavior (methods) related to one concept.

Why OOP in tests?

  • Selectors live in one file per page → UI changes touch one place
  • Tests read like English: cart.add(product) instead of page.locator('[data-id=...]').click()
  • Inheritance lets every page share logging, screenshot helpers, ad-blocking — without copy/paste

18. POM — Page Object Model

Each web page (or major component) is represented by a Python class. The class exposes user-level actions and state queries, never raw selectors.

Anatomy of a page object

class LoginPage(BasePage):
    URL = "/login"
    EMAIL_INPUT = '[data-qa="login-email"]'
    PASSWORD_INPUT = '[data-qa="login-password"]'
    LOGIN_BUTTON = '[data-qa="login-button"]'
    LOGIN_ERROR = "form p:has-text('incorrect')"

Why this works

  • Locators are constants at the top — easy to find and change
  • Methods return self when chainable: LoginPage(page).login(...).logout()
  • State queries (is_authenticated, is_on_login_page) make assertions readable
  • BasePage centralises wait helpers, logger, screenshot capture

Components vs pages

A component is a reusable widget that appears on multiple pages — site header, product card, price filter. Extracted so every page using them gets the same logic for free.

20. Playwright deep dive

Built-in tracing

Captures DOM snapshots, network, console, screenshots in one zip — viewable with npx playwright show-trace.

Locator strategy (in order of preference)

  1. page.get_by_role("button", name="Login") — accessibility-first, robust
  2. page.get_by_test_id("login-button") — explicit hooks via data-test
  3. page.get_by_text("Welcome") — user-visible text
  4. CSS attribute: [data-qa="login-button"]
  5. XPath — last resort. Mandated by the brief for product card collection in SearchResultsPage

21. context.route() — block ad domains at the network layer

Lets you intercept and control network requests before they reach the browser.

What "block at network layer" means

A page makes many requests — HTML, CSS, images, scripts, and ads. Ads come from external domains (doubleclick.net, googlesyndication.com, …). With context.route() we intercept and abort them before they load.

Code

context.route("**/*doubleclick.net*",      lambda route: route.abort())
context.route("**/*googlesyndication*",    lambda route: route.abort())
context.route("**/*facebook.com/tr*",      lambda route: route.abort())  # FB tracking pixel

Why we do this

ProblemSolution
Ads cause random popups that break testsBlock them — they never load
Ad scripts slow page loadBlock them — tests run faster
Ad networks are flaky (timeouts, errors)Block them — no external dependency
Cookie consent banners from ad trackersBlock the tracker — banner often disappears

22. Does Playwright auto-fix locators?

No. Playwright auto-waits for elements; it doesn't auto-fix broken selectors.

External tools that try

  • Playwright Codegen — records new selectors by watching you click
  • Healing locators (Healenium and similar) — try to find similar elements when the original fails

Bottom line: POM + robust locators (get_by_role, data-testid) minimise how often you need to update selectors.

23. Fixtures & conftest.py

Fixtures are the killer feature of pytest. They:

  • Set up resources (browser, page, login session)
  • Tear them down automatically
  • Have scopes: function (default), class, module, session
  • Inject into tests by parameter name

24. Secrets — three layers

  1. .env (gitignored) — plain text, local only. Loaded by python-dotenv.
  2. SOPS + age — optional. Encrypt .env at rest in the repo as secrets/.env.sops. Decrypts at runtime with the user's age key.
  3. gitleaks pre-commit hook — blocks commits that contain secrets (regex-matched).

25. AI tooling layer (MCP, Cursor rules, AGENTS.md)

MCP — Model Context Protocol

Open standard for connecting LLMs to external tools/data. Two servers wired in:

  • Playwright MCP — lets an AI agent drive the browser directly (record, replay, inspect)
  • Filesystem MCP — gives the agent scoped file read/write access

Cursor project-level rules

Files under .cursor/rules/*.mdc. Each rule is an "always-on" instruction or pattern (e.g. "never put hard-coded waits", "prefer get_by_role"). Cursor injects them into every prompt automatically.

Slash commands

.cursor/commands/*.md — reusable prompts triggered with /command-name, e.g. /add-page scaffolds a new POM class with the right boilerplate.

Personas

docs/personas/*.md — role descriptions (QA Architect, SDET, AI Reviewer) the user can invoke to shift the AI's tone and priorities.

AGENTS.md

Single source of truth for any AI assistant working in the repo: project conventions, naming rules, locator policy, security policy, do / never-do. Universal — works with Cursor, Claude, Copilot, Aider, etc.

26. CAPTCHA

"Completely Automated Public Turing test to tell Computers and Humans Apart" — a challenge proving you're a human, not a bot. Sites use it to block automated scripts (fake accounts, spam, scraping, brute-force).

Common types

TypeExampleHow it works
Text CAPTCHADistorted letters/numbersType what you see: "xK9pL2"
Image CAPTCHA"Select all traffic lights"Click the right images
reCAPTCHA v2"I'm not a robot" checkboxAnalyzes mouse movement, may show images
reCAPTCHA v3InvisibleScores behavior silently (0.0 = bot, 1.0 = human)
hCaptchaSimilar to reCAPTCHAPrivacy-focused alternative
Math CAPTCHA"What is 3 + 7?"Simple arithmetic
Audio CAPTCHAListen and typeAccessibility alternative

Why it's a problem for automation

Test:    Fill login form → Click submit
Website: "Prove you're human first" → Shows CAPTCHA
Test:    ??? → Fails

How to handle CAPTCHAs in testing

ApproachWhen to use
Test environment without CAPTCHABest option — ask devs to disable for test/staging
Allowlisted IP / accountTest account or CI server IP bypasses CAPTCHA
API tokens / cookiesSkip the UI login entirely, inject auth directly
CAPTCHA-solving servicesLast resort, costs money (2Captcha, Anti-Captcha)
Mock the CAPTCHA endpointIntercept and fake the response

In our project: automationexercise.com doesn't have CAPTCHA on the login page, so tests work fine.
